## 1. 项目是什么

本项目是第十五届中国软件杯 A组赛题（出题方：锐捷网络），目标是构建一个 **景区导览 AI 数字人系统**。

系统最终呈现为两个端：
- **游客交互端**（Web页面或小程序）：游客打开页面后，会看到一个数字人形象（类似虚拟导游），可以通过语音或文字与数字人对话。数字人能回答景区相关问题（历史、文化、景点介绍）、根据游客兴趣推荐游览路线、说话时口型同步且有表情变化，体验上接近一个真人导游在跟你聊天。
- **管理后台**（Web页面）：景区管理员登录后，可以上传和管理知识库文档（这些文档就是数字人的"知识来源"）、配置数字人的外观和声音、查看数据大屏（今日服务人次、热门问题排行、游客满意度趋势等）、查看游客反馈分析报告。

### 游客端具体体验流程

1. 游客打开页面，屏幕中央显示数字人形象（2D半身像），底部有聊天输入框和语音按钮
2. 游客输入文字或按住语音按钮说话 → 系统识别意图 → 从景区知识库检索相关内容 → 大模型生成回答 → 数字人用语音回答，同时嘴巴动、表情配合
3. 游客可以说"我对历史感兴趣" → 数字人推荐一条历史文化游览路线，并可以沿路逐点讲解
4. 整个过程数字人有情感变化：回答开心的内容时微笑，回答严肃历史时表情庄重

### 管理后台具体功能

1. **知识库管理**：上传景区文档（PDF/Word/文本），系统自动解析并向量化，作为数字人的知识底座
2. **数字人形象管理**：上传数字人形象图片、选择/试听语音音色、预览效果
3. **对话记录**：查看历史对话，了解游客都问了什么
4. **游客反馈分析**：自动对对话做情感分析，生成满意度报告
5. **数据大屏**：服务人次、热门问答Top10、满意度趋势图、实时对话滚动展示

### 评分标准（决定开发优先级）

| 评分项 | 分值 | 说明 |
|---|---|---|
| 功能完整度 | 40分 | 所有功能能跑通且稳定 |
| 技术与创新性 | 30分 | 数字人表现力（15分）+ 大模型知识库准确度（15分），90%准确率硬指标 |
| 行业体验 | 20分 | 交互自然度、口型同步、情感表达，专家主观评估 |
| 文档质量 | 10分 | 设计文档 + PPT + 演示视频 |

**关键结论：** 功能完整是基本盘（40分），数字人效果和RAG准确率是拉开差距的核心（30分），体验自然度决定20分。先保证功能全跑通，再打磨数字人和准确率。

## 2. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 前端 | Vue 3 + Element Plus + ECharts | 游客端 + 管理后台 |
| Java后端 | Spring Boot + MyBatis-Plus + MySQL | 面向前端的唯一网关，管理所有业务数据（CRUD、用户、会话） |
| Python后端 | FastAPI | AI能力提供者：RAG问答、ASR、TTS、情感分析、数字人驱动 |
| 向量数据库 | ChromaDB | 存储景区知识向量，Python端直接调用 |
| 大模型 | 小米 MiMo API（mimo-v2.5-pro）| 问答推理，OpenAI兼容格式调用 |
| 语音识别 | FunASR（阿里开源） | 中文ASR，Python端部署 |
| 语音合成 | CosyVoice（阿里开源） | 中文TTS，支持音色定制 |
| 数字人驱动 | SadTalker 或 MuseTalk | 2D数字人，输入图片+音频 → 生成口型同步视频 |
| 部署 | Docker + Docker Compose | 三服务统一打包 |

## 3. 系统架构

```
游客端(Vue3)  ←HTTP/WS→  Java后端(Spring Boot)  ←HTTP→  Python后端(FastAPI)
                                ↕                              ↕
                              MySQL                      ChromaDB / 大模型API
                                                    FunASR / CosyVoice / SadTalker
```

核心原则：
- **Java后端**是面向前端的唯一入口。前端不直接访问Python。Java管理所有业务数据、用户会话、文件存储。
- **Python后端**只做AI。Java把"用户说了什么"转发给Python，Python返回"AI的回答+音频+视频数据"，Java再转给前端。
- **前端**通过 WebSocket 保持与Java的实时对话通道。数字人渲染在前端完成（接收音频/视频数据播放）。
- 两个后端通过 REST API 通信，接口格式在每个Phase开始时对齐。

## 4. 开发路线图

### Phase 0：项目骨架搭建与最小链路跑通

**目标：** 三端骨架就绪，跑通"文字输入 → 大模型回答 → 文字显示"全链路。

**要做的事：**
- 搭建 Git 仓库（main + dev + 三人各自分支）
- 初始化 Vue3 项目：基础路由（游客端/管理后台两个布局）、一个简单的聊天页面（输入框+消息列表）
- 初始化 Spring Boot 项目：引入基础依赖，一个聊天接口（接收前端消息，转发给Python，返回结果）
- 初始化 FastAPI 项目：接入大模型API（Qwen或GLM），一个最简问答接口（接收问题 → 调API → 返回文字回答）
- 数据库建初始表结构（用户表、景点表、对话记录表、知识文档表——先粗后细，后续迭代）

**里程碑：** 页面输入"故宫几点开门"，数字人返回文字回答。

---

### Phase 1：RAG 知识库构建

**目标：** 数字人能基于景区资料精准回答事实性问题，准确率 ≥ 90%。

**要做的事：**
- 获取并解析示范景区资料包（官方提供的PDF/Word/文本文件），提取纯文本
- 文档切片：按段落或语义切成 300-500 字的 chunk，保留上下文重叠（前后各50字）
- 用 Embedding 模型（BGE系列）把每个 chunk 向量化，存入 ChromaDB
- 实现 RAG 检索链路：用户问题 → Embedding → ChromaDB 检索 Top-K 相关片段 → 拼入 System Prompt（附带检索到的原文）→ 调大模型生成回答
- Prompt 中强制要求"只根据提供的资料回答，资料中没有的明确说不知道"，防止幻觉
- 建一个标准测试集（至少20个景区事实性问题+标准答案），自测准确率
- 管理后台实现知识库管理页面：文档上传、列表、删除，上传后触发Python端重建向量索引

**里程碑：** 标准测试集 20 个问题，至少 18 个回答正确。

---

### Phase 2：语音交互链路

**目标：** 游客可以语音提问，数字人语音回答。语音问答全链路延迟 < 5 秒。

**要做的事：**
- Python端集成 FunASR：接收音频数据 → 转文字 → 送入 RAG 问答链路
- Python端集成 CosyVoice：大模型回复文字 → 生成自然语音音频
- Python端封装统一对话接口：接收音频或文字 → 自动判断输入类型 → ASR（如果是音频）→ RAG问答 → TTS → 返回文字回答 + 音频数据
- 支持流式输出（SSE或WebSocket）：大模型边生成文字边送TTS，不等全部生成完再合成，减少体感延迟
- 前端实现录音功能（MediaRecorder API）、语音播放、对话气泡UI（区分用户消息和数字人消息）
- Java后端用 WebSocket 管理实时对话会话，转发音频/文字数据

**里程碑：** 按住说话 → 数字人语音回答，端到端延迟 < 5秒。

---

### Phase 3：数字人形象与口型同步

**目标：** 数字人形象出现在屏幕上，说话时嘴型同步动、有表情变化。

**要做的事：**
- 选定数字人形象图片（可用景区IP形象、AI生成的导游照片、或真人照片）
- 集成 SadTalker/MuseTalk：输入形象图片 + TTS音频 → 生成口型同步的数字人说话视频
- 设计2-3个预设表情状态（默认微笑、开心、思考），在Python端加入情感判断：分析大模型回复的情绪倾向 → 输出表情标签 → 前端切换对应表情/视频
- 优化生成速度：降低输出分辨率、用GPU加速、或用 MuseTalk（实时性更好）
- 前端数字人展示区域：视频播放器（自动播放、循环播放）、根据表情标签切换视频
- 管理后台：数字人形象管理页面（上传形象图片、选择/试听语音音色）

**里程碑：** 数字人在屏幕上说话，嘴型自然、有表情切换，整体不违和。

---

### Phase 4：个性化路线推荐

**目标：** 游客表达兴趣偏好后，数字人推荐匹配的游览路线并逐点讲解。

**要做的事：**
- 设计兴趣标签体系（历史、自然风光、亲子、摄影、美食等），给数据库中每个景点打标签
- 预存几条不同主题的推荐路线模板（写入知识库），走 RAG 检索
- 推荐逻辑：游客说"我对历史感兴趣" → 识别意图 → 检索匹配的历史路线模板 → 大模型结合景点信息生成个性化推荐话术 → 数字人语音播报
- 游客端增加兴趣选择入口（快捷标签按钮 + 文字输入皆可）
- 游客端展示推荐路线（列表或地图形式均可）

**里程碑：** 游客说"喜欢自然风光"，数字人推荐一条自然景观路线并逐个讲解。

---

### Phase 5：管理后台完善 + 数据大屏

**目标：** 管理后台功能完整，数据大屏展示核心运营数据。

**要做的事：**
- 对历史对话记录做情感分析（正面/中性/负面），统计结果存入数据库
- 提取高频问题排行、游客关注热点
- 数据大屏页面（ECharts）：当日/本周服务人次、热门问答Top10、满意度趋势折线图、实时对话滚动窗口、游客兴趣画像饼图
- Java后端实现聚合查询接口（按天/周统计，情感分布，热门问题等）
- 管理后台其他页面补全：对话记录查看、游客反馈列表

**里程碑：** 管理员打开后台，看到完整运营数据大屏，数据与实际交互一致。

## 5. 关键风险

| 风险 | 应对 |
|---|---|
| 数字人生成太慢（>5秒） | 降低分辨率、用MuseTalk替代SadTalker、预生成常见回答的视频缓存 |
| RAG准确率不到90% | 优化切片策略、增加FAQ精确匹配层、Prompt工程反复调优、扩充测试集 |
| 语音全链路延迟超标 | 流式TTS（边生成边合成）、音频分段传输、不做等全句生成完再合成 |
| 前端流式渲染问题 | Phase 2就引入WebSocket/SSE，不要拖到最后 |
