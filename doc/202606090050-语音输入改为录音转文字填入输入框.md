# 语音输入改为：录音→转文字→填入输入框

## 提示词

> 现在新的问题是语音输入后用户这边显示🎤🎤 正在识别...，但是ai这边回复抱歉，消息格式错误。我希望把这个功能大改一下，语音输入变成语音转文字，内容输出到输入框里里面，待用户修改后再发出去

## 做了什么

### 1. 改动思路

原来的语音流程：录音 → WebSocket 发音频 → Java 转发 → Python ASR → RAG → LLM → TTS → 语音回复
问题：链路太长，容易出错，且用户无法控制回复内容。

新流程：录音 → 直接调 Python ASR → 文字填入输入框 → 用户编辑 → 点发送

### 2. 具体改动

**前端（4 个文件）：**

| 文件 | 改动 |
|------|------|
| `vite.config.js` | 新增 `/py-api` 代理 → Python 8000（绕过 Java 直调 ASR） |
| `chat.js` (API) | 新增 `recognizeSpeech(blob, format)` 函数，POST 到 `/py-api/asr` |
| `ChatInput.vue` | 重写：录音停止后调 ASR，结果填入 `inputText`，按钮文字改为"语音输入"/"停止录音"/"识别中..." |
| `ChatView.vue` | 去掉 `@send-voice` 事件和 `handleSendVoice` 函数 |

**交互流程：**
1. 点击「语音输入」→ 开始录音（红色波浪动画 + 计时）
2. 点击「停止录音」→ 调用 ASR 识别（按钮显示"识别中..."）
3. 识别结果填入输入框 → 自动聚焦
4. 用户可编辑文字 → 点击「发送」

### 3. 验证

- ✅ `npm run build` — built in 799ms
- ⚠️ ASR 模型首次调用需下载（约 1-2GB），后续调用秒级响应

## 为什么这样做

原来的 WebSocket 音频流链路太长（前端→Java→Python ASR→RAG→LLM→TTS→Java→前端），任何一环出错都会导致"消息格式错误"。改为直接调 ASR 端点 + 填入输入框，用户可以确认/编辑识别结果后再发送，体验更好。

## 产生的结果

语音输入改为"录音→转文字→填入输入框→用户编辑→发送"的两步式交互。
