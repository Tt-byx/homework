# 修复 WebSocket 转发：输入为空和卡住问题

## 提示词

> http://localhost:5173/chat在这里面，输入文本后按回车会之间返回"抱歉，输入为空"，如果输入文本后点击发送按钮后，会出现数字人正在思考的动画，但是似乎卡住了，一直不回答

## 做了什么

### 1. 问题诊断

通过端到端测试定位到两个问题的根因：

**问题 1："输入为空"** — Java WebSocket handler 用 `HttpClient` 发 multipart/form-data 到 Python `/api/chat/stream`，但 Python 解析 multipart body 失败，`message` 参数为 null。

**问题 2："卡住不回答"** — 同样是 Java 转发失败，但错误消息没有正确回传给前端，导致 loading 状态永远不结束。

### 2. 修复方案

**核心改动：** `ChatWebSocketHandler.java` 的 `handleTextChat` 方法

| 修改前 | 修改后 |
|--------|--------|
| `HttpClient` 发 multipart 到 `/api/chat/stream` | `RestTemplate` 发 JSON 到 `/api/chat` |
| 手动构建 multipart body（格式不兼容） | Spring 自动处理 JSON 序列化 |
| SSE 响应解析 | 直接返回完整 JSON 响应 |

**同时修复：**
- `proxyAudioToPython` 也改用 `RestTemplate`（multipart 由 Spring 自动处理）
- 音频消息的 `session_id` 和 `format` 正确传递
- 删除不再使用的 `buildMultipartBody` 和 `buildMultipartAudioBody` 方法

### 3. 验证结果

```
WebSocket 连接 → 发送消息 → Python RAG 检索 → MiMo 回答 → TEXT 回复 → DONE
```

三端编译/构建/启动全部通过：
- ✅ Java `mvn clean compile` — BUILD SUCCESS
- ✅ Python `uvicorn` — 启动成功
- ✅ 前端 `npm run build` — built in 905ms
- ✅ WebSocket 端到端测试 — 收到回复和完成标记

## 为什么这样做

Java 的 `HttpClient.BodyPublishers.ofString()` 发送 multipart body 时，Python 的 `python-multipart` 库无法正确解析（body 被识别为 null）。改用 Spring 的 `RestTemplate` 发 JSON 彻底绕过了这个问题。

## 产生的结果

文字聊天通过 WebSocket 全链路跑通。用户在 /chat 页面输入文字后，可以通过 WebSocket 发送并收到 AI 回答。
