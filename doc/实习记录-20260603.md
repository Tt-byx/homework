## 一、工作进展

**本次完成模块：** Vue 3 前端搭建、Python 后端搭建、全链路联调

**完成内容：**

1. Vue 3 前端项目初始化
   使用 Vite 创建 Vue 3 项目，安装 vue-router、pinia、axios、element-plus 等依赖
   实现双布局路由：游客端（顶部导航 + 内容区）和管理后台（左侧菜单 + 内容区）
   聊天页面核心组件：
   - `ChatMessage.vue`：消息气泡组件，区分用户消息（右对齐蓝色）和 AI 回复（左对齐灰色）
   - `MessageList.vue`：消息列表，自动滚动到底部，空状态提示
   - `ChatInput.vue`：输入框组件，支持 Enter 发送、Shift+Enter 换行、发送中 loading 状态
   - `ChatView.vue`：聊天主页面，组合以上三个组件

   使用 Pinia 管理聊天状态（消息列表、会话ID、loading 状态）
   axios 封装统一请求层，配置 Vite proxy 转发 `/api` 到 Java 后端 8080 端口，解决开发环境跨域问题

2. Python 后端（FastAPI）搭建
   使用 uv 创建项目并安装依赖：fastapi、uvicorn、openai（OpenAI SDK）、pydantic-settings
   项目结构：
   - `app/core/config.py`：使用 pydantic-settings 从 `.env` 文件读取 API 密钥等配置
   - `app/schemas/chat.py`：Pydantic 请求/响应模型，自动做参数校验
   - `app/services/llm_service.py`：封装大模型 API 调用，使用 OpenAI SDK 的兼容模式连接小米 MiMo API
   - `app/routers/chat.py`：`POST /api/chat` 路由，接收消息并调用大模型返回回复

   大模型接入方式：小米 MiMo API 兼容 OpenAI 协议，通过 `AsyncOpenAI` 客户端指定自定义 `base_url` 即可调用，无需额外 SDK

3. 全链路联调
   三个服务全部启动后，通过浏览器在聊天页面输入问题，验证完整数据流：
   前端(5173) → Vite proxy → Java 后端(8080) → MySQL 写入会话和消息 → Python 后端(8000) → MiMo API → 原路返回 → 页面显示 AI 回复

   联调过程中发现的问题：
   - Pydantic 字段命名风格：Python 端用 `session_id`（下划线），Java 端用 `sessionId`（驼峰），后续需要注意两端字段对齐
   - uvicorn 进程在 Windows 上需要 `taskkill` 强制关闭，`Ctrl+C` 有时杀不干净导致端口占用

**完成百分比：** 整体项目约 **25%**（Phase 0 骨架搭建全部完成，下一步进入 Phase 1 RAG 知识库构建）

## 二、存在的问题

1. Pydantic 请求体解析报 400
   最初用 curl 测试 Python `/api/chat` 接口时返回 400 Bad Request，排查后发现不是代码问题，而是 Git Bash 环境下 curl 对 JSON 中文和特殊字符的转义处理有问题，改用 Python httpx 发请求后正常。

2. 大模型 API 返回内容的编码问题
   MiMo API 回复中包含 emoji 表情，在 Windows 终端输出时遇到 GBK 编码报错（`UnicodeEncodeError: 'gbk' codec can't encode character`）。这不是功能 bug，只是终端显示问题，数据本身传输正确。
