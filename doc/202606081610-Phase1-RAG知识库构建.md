## 二、为什么这样做

### 架构决策
1. **Python 内部完成 RAG**：Java ChatServiceImpl 无需改动，Python 的 `/api/chat` 端点内部执行 embedding → 检索 → 注入 prompt → 调用 LLM。最小化改动面。
2. **单 ChromaDB Collection**：所有文档共用 `scenic_knowledge` 集合，metadata 存 doc_id/doc_title，删除时按 doc_id 过滤。简单高效。
3. **Java 异步调用 Python**：上传文件后立即返回，Python 后台处理文档（解析→切片→向量化），前端轮询状态。
4. **ChromaDB 本地持久化**：Docker 方案因网络问题放弃，PersistentClient 方式更简单可靠。

### RAG Prompt 设计
- System Prompt 包含防幻觉指令：「只根据参考资料回答，资料中没有的明确说不知道」
- 检索结果带来源文档标题，增强可信度
- 距离阈值 0.7，过滤不相关的检索结果

## 四、已知问题与后续优化

1. **Embedding API 延迟**：SiliconFlow API 响应较慢（2-5秒），影响聊天体验。后续可考虑本地部署 BGE 模型。
2. **知识库覆盖**：部分问题（表演时间、设计师）文档中未提及，需要补充更多景区资料。
3. **Windows 终端编码**：中文字符在 Git Bash 中显示为乱码，不影响实际功能。
